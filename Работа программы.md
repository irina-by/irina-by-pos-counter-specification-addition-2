Как распознаются слои в программе?

Два отдельных блока: сначала только палитра (выноски), потом только спецификация (наименование + «Кол.») с примерами.

Часть 1. Только палитра (кнопка ЗАПУСТИТЬ)
Что это вообще
Палитра после ЗАПУСТИТЬ показывает подсчёт выносок на плане: какая марка (номер позиции), сколько раз встретилась, на каком слое лежит текст.

Чертеж при этом не меняется — только таблица в палитре.

Откуда берутся данные
Программа смотрит объекты, которые вы ей «дали»:

Как нажали ЗАПУСТИТЬ	Что сканируется
На чертеже уже выделены объекты
Только выделение (галочка «Все объекты» не важна)
Галочка «Все объекты в модели» включена, выделения нет
Вся модель или объекты в активном видовом экране (если вы на листе)
Ничего не выделено, галочка выключена
Подсчёт не идёт — нужно выделить или включить галочку
Из объектов читаются тексты (TEXT, MTEXT) и блоки (атрибуты, вложенные тексты). MLeader на этом этапе не используется.

Как узнаётся марка (номер позиции)
Программа ищет число в тексте выноски:

Сначала отрезает приставки вроде: Поз., Позиция, Марка, №, POS, Item и т.п.
Оставшееся должно быть только цифрами (без точек, запятых, букв).
Число от 1 до 10000 → это марка (внутри — поле Key).
Примеры:

Текст на выноске	Марка	Примечание
52
52

52
Приставка срезается
Марка 3

не марка

поз. А-52
не марка
Есть буквы

не марка
Нет известной приставки, в тексте не только цифры
Если марку вытащить не удалось, строка всё равно может попасть в палитру как обычный текст (без номера для связи со спецификацией).

Как считается количество
Одинаковые выноски складываются в одну строку:

Группировка по паре: слой + текст (после нормализации).
Каждая найденная выноска → +1 к количеству.

Как определяется слой в палитре
Слой берётся из объекта в AutoCAD, на котором лежит текст:

У обычного текста — его слой.
У текста в блоке со слоем «0» — подставляется слой вставки блока (как в AutoCAD).
Если имя слоя вида БЛОК|ВЫНОСКИ — в палитре показывается часть после | → ВЫНОСКИ.
Важно: для палитры слой не угадывается статистикой — это фактический слой выноски.

Колонка «Наименование» в палитре (до спецификации)
После одного только ЗАПУСТИТЬ колонка наименования пустая (или «не найдено»).

Имя появляется после «Выбрать спецификацию» — см. часть 2.

Фильтры в палитре (только отображение)
Внизу таблицы можно отфильтровать строки по:

тексту / марке,
наименованию,
количеству,
слою.
Фильтры не пересчитывают чертёж и не меняют подсчёт в памяти — только скрывают строки в таблице.

Особый случай — фильтр по слою и имена из спецификации:

Если вы уже выбрали спецификацию и потом отключили слой выноски в фильтре — для этих строк наименование сбрасывается (чтобы имя из таблицы не «прилипло» к выноске другого слоя).

Пример:

ЗАПУСТИТЬ → марка 52, слой ВЫНОСКИ, количество 5.
Выбрать спецификацию → в наименовании: Труба стальная Ду50.
В фильтре слоёв сняли галочку с ВЫНОСКИ → строка 52 скрыта; если снова показать — имя вернётся.
Что уходит в спецификацию из палитры
Когда нажимают «Выбрать спецификацию», из палитры для записи «Кол.» берётся:

марка (Key),
сумма количества по видимым строкам палитры (с учётом фильтров).
Если марки 52 в палитре нет или количество 0 — в «Кол.» для 52 ничего не запишется (будет сообщение «нет количества для марок…»).

Типовой сценарий (только палитра)
1. NETLOAD → палитра открылась
2. Выделили выноски на плане (или включили «Все объекты»)
3. ЗАПУСТИТЬ
4. В таблице: Марка | Количество | Слой | (Наименование пусто)
5. При необходимости — фильтр по слою, экспорт внизу палитры
Чертеж не трогается.

Часть 2. Только спецификация (наименование + «Кол.»)
Что это
Кнопка «Выбрать спецификацию»:

Вы рамкой выбираете таблицу (линии + текст). Можно несколько таблиц подряд; Enter без рамки — конец выбора.
Программа строит сетку ячеек по линиям.
Находит столбцы по заголовкам в шапке (марка, наименование, количество…).
Подставляет наименования в палитру (по номеру марки).
Пишет цифру количества в колонку «Кол.» на чертеже (из палитры после ЗАПУСТИТЬ).
На чертеже меняется только «Кол.» — наименование в ячейке таблицы программа не переписывает, только показывает в палитре.

Шаг 0. Что должно быть до выбора спецификации
Условие	Что будет
Был ЗАПУСТИТЬ, в палитре есть марки и количества
И имена, и «Кол.» могут заполниться
ЗАПУСТИТЬ не делали
Имена могут появиться, «Кол.» на чертеже — нет (неоткуда взять количество)
Как строится таблица (сетка)
В рамке читаются LINE (линии) и TEXT/MTEXT (текст).
Слой сетки — слой, на котором больше всего линий таблицы (авто).
По линиям считаются вертикали и горизонтали → ячейки.
Каждый текст попадает в ячейку по координатам (с небольшим допуском ~2 мм). Для **MText и DBText (TEXT)** в шапке и данных берётся **центр рамки текста** (GeometricExtents), а не угол вставки — иначе «Поз.» / «Кол.» с выравниванием Top Left могли попадать в соседний столбец. Шапка читается и по ячейке сетки, и по геометрии X/Y в полосе столбца.

**Многострочная шапка СПДС:** заголовки «Поз.» / «Кол.» часто в верхней части таблицы, но линии сетки могут быть на разных слоях и давать неверные строки. **Сначала** программа ищет слова в **верхних 2000 мм** по координатам текстов (не по линиям): от самой верхней надписи («Спецификация») вниз до строки с «Поз.» / «Кол.» / «Наименование», по X определяет столбцы. Затем **проверяет столбец «Поз.»**: если в нём меньше двух цифр марок в данных таблицы — переключает на столбец, где цифры есть (чтобы не путать с «Таблица 1» в шапке). Если не нашла — запасной способ по сетке ячеек. При сбое в CMD: «Шапка по текстам…» и «Столбец «Поз.» уточнён: …».
Пример: выделили рамкой только одну спецификацию на листе — программа обрабатывает её как отдельную таблицу (scope 0). Вторую рамку — scope 1 и т.д.

Распознавание столбцов (по шапке)
По тексту в верхних строках ищутся заголовки (подстрока в ячейке):

Столбец	Обычные слова в шапке
Марка
«марка», «поз», «№»…
Наименование
«наимен», «назван»…
Кол.
«кол», «колич»…
Если столбец не найден — эта функция для него не работает (нет ColName → нет имён; нет ColQty → нет записи в «Кол.»).

**Важно:** если в CMD видно `столбец 0 «—»`, `столбец 1 «—»` — это **не** успешное распознавание, а ошибка (раньше программа назначала столбцы при пустой шапке). Сейчас без совпадения слова в заголовке будет «не найдена» и отдельное предупреждение `[POSC]`.

Столбцы назначаются **без дублирования**: сначала фиксируется Марка (лучший score), затем Кол. (лучший среди оставшихся), затем Наименование. Если один заголовок подходит под несколько ролей — побеждает та, что назначается раньше; остальные ищут следующий столбец.

Слои в спецификации (кратко, с примерами)
A. Слои основного текста таблицы
Программа смотрит все тексты в ячейках данных и шапки:

Слои, на которых лежит ~90% текста в столбцах спецификации → «разрешённые» (марка, имя, кол. и т.д.).
Редкие слои (сноски, пометки инженера) → «исключённые», их текст игнорируется.
Пример 1 — нормально:

Слой	Где текст	Доля
СПЕЦ_ТЕКСТ
наименования, марки
~95%
СПЕЦ_ПРИМ
«см. узел», «*» в углу ячейки
~5%
→ Читается только СПЕЦ_ТЕКСТ, примечания отбрасываются.

Пример 2 — проблема:

Половина наименований на СПЕЦ_ТЕКСТ, половина на СПЕЦ_ИМЯ2 без явной закономерности → второй слой может попасть в «лишние», часть имён пропадёт.

B. Слой наименования (отдельно для столбца «Наименование»)
В столбце наименования:

Считается, на каком слое больше всего строк → главный слой наименования.
Другие слои в том же столбце:
если ≥ ~5% текстов и строки похожи на нормальное имя (не просто цифра) → дополнительный слой (многострочное имя на другом слое);
иначе → пометка, не имя.
Пример 3 — многострочное имя:

→ Оба фрагмента собираются в одно имя для марки.

Пример 4 — сноска в той же ячейке:

Как собирается наименование для марки
Для каждой марки (например 52):

Находится строка марки в столбце «Марка».
Берутся все строки наименования от этой позиции до следующей марки (53, 54…), даже если ячейка марки объединена на несколько строк по вертикали.
Из ячеек столбца «Наименование» собираются части текста (с учётом слоёв из пункта B).
Склеиваются в одну строку → попадают в палитру в колонку Наименование для марки 52.
Пример 5 — марка 52 на двух строках сетки:

| Марка | Наименование              |
|   52  | Труба стальная бесшовная |
|       | Ду 50, стенка 3 мм       |
|   53  | Отвод 90°                |
→ Для 52 в палитре: Труба стальная бесшовная Ду 50, стенка 3 мм
→ Для 53: Отвод 90°

Пример 6 — связь с палитрой:

Палитра (после ЗАПУСТИТЬ)	Спецификация
Марка 52, кол. 5
В строке марки 52 имя Труба…
Марка 53, кол. 2
В строке 53 имя Отвод 90°
После выбора спецификации в палитре у строк 52 и 53 заполняется Наименование.

Пример 7 — фильтр слоя в палитре:

Выноска 52 на слое ВЫНОСКИ_СХЕМА, а в фильтре включён только ВЫНОСКИ → имя для 52 не покажется, хотя в спецификации оно есть.

Как записывается «Кол.» на чертеже
Из палитры берётся словарь: марка → суммарное количество (по видимым строкам).
Для каждой марки в таблице находится ячейка столбца «Кол.» в той же строке, что и марка.
В центр ячейки ставится или обновляется текст с числом.
Стиль цифры «Кол.»:

Как основной текст таблицы (слой/шрифт/цвет с колонок наименования или марки),
не как мелкая сноска инженера в ячейке «Кол.».
Пример 8 — запись количества:

Палитра: марка 52 → количество 5.
В спецификации в строке 52 в «Кол.» пусто или старая цифра → появится/обновится 5 в стиле таблицы.

Пример 9 — примечание инженера не удаляется:

В ячейке «Кол.» уже есть MText: 5 и ниже сноска * уточнить на объекте.
Программа старается обновить основную цифру; полностью стереть примечание не обязана — сноска может остаться рядом (как в инструкции).

Пример 10 — нет количества в палитре:

В спецификации есть марка 99, в палитре после ЗАПУСТИТЬ марки 99 не было → в «Кол.» для 99 не пишется, в статусе: «Нет количества для марок: 99».

Пример 11 — несколько таблиц:

Две спецификации на листе — две рамки.
Количество 5 для марки 52 из палитры запишется в обе таблицы, если в каждой есть строка с маркой 52 (каждая таблица — свой scope, стиль «Кол.» свой).

## Замечание: ручная таблица из Line, огромные цифры, «Кол.» не в колонке

**Наблюдение (тестовый чертёж, июнь 2026):** таблица собрана вручную из отрезков Line (Поз. / Обозначение / Наименование / Кол. / Масса). Заголовки правильные. В колонке «Поз.» в строках данных пусто. После «Выбрать спецификацию» программа вставляет TEXT с количеством **не в ячейку «Кол.»**, а в другое место; высота текста **250** — визуально огромная относительно таблицы.

Пример вставленного текста из Properties:
- Contents: `1` (или `2`)
- Height: **250**
- Style: GOST-2.304_Type-B
- Layer: `0` или `TM_Текст` (как у образца текста таблицы)
- Justify: Middle center
- Text alignment: X ≈ 483, Y ≈ 191 (при том что строки таблицы лежат примерно на Y 275–362)

Линии таблицы:
- **Вертикальные** — Line, слой **TM_OCH** (длина ~270 ед. чертежа).
- **Горизонтальные** — Line, но на **разных слоях** (например одна на слое `0`, другая на `TM_OCH`; длина ~185 ед.).

### Высота текста «Кол.» (после исправления spec-grid-qty-fix)

**Было:** константа `QtyTextHeight = 250.0` — всегда огромные цифры на маленьких таблицах.

**Стало:** высота берётся из текста спецификации (`QtyTableTextAppearance.TextHeight`):
1. Шапка и данные колонки **«Кол.»**
2. Тело таблицы (**Наименование** / **Поз.**)
3. Запас: `QtyTextHeightFallback = 2.5`

Стиль GOST и слой TM_Текст — как у образца из таблицы (приоритет ColQty).

### Почему цифра не попадает в колонку «Кол.» (логика кода, без правок)

Программа **не ищет** пустую ячейку «Кол.» глазами. Она:

1. Строит координатную сетку **GridXs / GridYs** только по отрезкам **Line** из рамки выбора.
2. По тексту в **первых двух строках** сетки находит индексы столбцов: ColMark («поз»), ColName («наимен»), ColQty («кол»).
3. Привязывает марки из **колонки «Поз.»** (только чистые цифры 1, 2, 3…).
4. Ставит количество в точку:
   - **X** = середина между `GridXs[ColQty]` и `GridXs[ColQty + 1]`
   - **Y** = середина между `GridYs[rowTop]` и `GridYs[rowTop + 1]`

Если сетка или привязка марок неверны — координаты «уезжают», хотя заголовок «Кол.» на экране правильный.

#### Причина 1. Горизонтальные линии на разных слоях

Автовыбор слоя сетки (`AutoDetectGridLayer` в `TableGrid.cs`):

- Считаются кандидаты среди линий (см. причину 2 про длину).
- Берётся слой, на котором **больше всего** таких линий.
- Если его доля **≥ 30%** — сетка строится **только с линий этого слоя** (`FilterLines`).
- Остальные слои **отбрасываются**.

На тестовой таблице:
- вертикали на **TM_OCH**;
- горизонтали частично на **`0`**, частично на **TM_OCH**.

Если «победит», например, слой `0` — в сетку попадут горизонтали `0`, но **вертикали TM_OCH исчезнут** → столбцы схлопнутся или сместятся, **ColQty указывает не на ту координату X** (цифра оказывается слева, ~X 483, а не в «Кол.»).

Если «победит» **TM_OCH** — наоборот, теряются горизонтали слоя `0` → **не хватает горизонтальных границ строк**, **GridYs** неверны, **Y** уходит далеко от таблицы (как Y ≈ 191 при таблице на Y 275–362).

Если доли слоёв близки (**< 30%** у лидера) — в лог пишется `auto grid layer ambiguous → no-layer-filter`, и тогда берутся **все** линии. Это лучше для смешанных слоёв, но пользователь **не видит** явного предупреждения в командной строке (диагностика TABLE-GRID в CMD по умолчанию отключена).

#### Причина 2. Порог MinGridLineLen = 5000

В `TableGrid.cs`: `MinGridLineLen = 5000.0`.

Этот порог используется **только** при выборе слоя сетки (`IsGridCandidate`), **не** при отборе линий в саму сетку.

На маленькой ручной таблице длина линии ~185 (горизонт) и ~270 (вертик) **меньше 5000** → **ни одна линия не кандидат** → слой сетки = `null` → фильтр по слою **не применяется**, в сетку идут все Line.

**Вывод для этого чертежа:** сбой «не в Кол.» здесь сильнее связан не с отсечением коротких линий из сетки, а с **геометрией сетки при смешанных слоях** (когда кандидаты всё же есть на других чертежах) и с **пустой «Поз.»** (ниже). На данном тесте при чисто коротких линиях слои формально не фильтруются — тогда ошибка может давать **лишняя длинная линия в рамке выбора** (на скрине видна вертикаль, уходящая далеко вниз) или **неверная кластеризация осей** при неполном наборе линий в рамке.

#### Причина 3. Пустая колонка «Поз.»

Привязка марки к строке (`BindKeys`, `TryParseMarkKey`):
- в «Поз.» должны стоять **только цифры** (`1`, `2`, `3`…);
- `Ушко 1`, `21 OCT…` **не** считаются маркой;
- пустая ячейка — марки нет.

Тогда `KeyToRowTopSub` пустой → по коду **запись «Кол.» для строк таблицы не должна идти** (цикл по ключам пустой).

Если цифра всё же появилась — возможны варианты:
- в палитре есть марки после ЗАПУСТИТЬ, а в таблице для них нашлась привязка (цифра где-то попала в полосу X колонки «Поз.» из-за сдвинутой сетки);
- в рамку попали **другие** тексты/линии вне таблицы, сдвинувшие GridXs/GridYs;
- на чертеже остался текст от **предыдущего** запуска.

**Практическое правило:** для записи «Кол.» в строке таблицы должны быть марки **именно в колонке «Поз.»** (чистые числа), и они должны совпасть с марками в палитре после ЗАПУСТИТЬ.

#### Причина 4. Заголовок правильный — но это не гарантия координат

Заголовок «Кол.» читается для **номера столбца** ColQty. Координата вставки берётся из **линий**, а не из положения надписи «Кол.». Заголовок может быть верным, а **GridXs[ColQty]** — нет, если вертикали не попали в сетку или оси сдвинуты.

### Сводка симптомов на тестовой таблице

| Симптом | Вероятная причина по коду |
|--------|---------------------------|
| Цифра огромная (Height 250) | Константа `QtyTextHeight = 250.0`, не масштаб ячейки |
| Не в колонке «Кол.» | Неверные GridXs → неверный ColQty по X |
| Y далеко от строк таблицы | Неверные GridYs (не все горизонтали в сетке / лишние линии в рамке) |
| Слой/стиль как у таблицы | Образец берётся из текста наименования (TM_Текст, GOST…) — штатно |
| Middle center | `ApplyQtyCenterAlignment` — центр расчётной ячейки |
| «Поз.» пустая | Нет привязки марок → «Кол.» не должна заполняться по строкам |

### Исправление (spec-grid-qty-fix, реализовано)

1. **Смешанные слои линий** — `BuildMergedGridAxes`: доминирующий слой (≥30%) + виртуальное дополнение осей с других слоёв. CMD: `[POSC] Сетка таблицы: линии на разных слоях — оси дополнены (TM_OCH + 0).`
2. **Высота «Кол.»** — из текста таблицы, не 250.
3. **Центр «Кол.»** — X по геометрическому центру ячейки сетки (`gridCenter`); Y по строке данных сетки.
4. **Пустая «Поз.»** — CMD: `[POSC] Заполните колонку «Марка» (Поз.) в таблице — иначе «Кол.» не запишется.`
5. **Сетка не построилась** — CMD: `[POSC] Не удалось построить сетку таблицы — проверьте линии в рамке выбора.`

План: `.cursor/plans/spec-grid-qty-fix_94cefd16.plan.md`

### Что проверить на такой таблице (без изменения кода)

1. Все **горизонтальные и вертикальные** линии сетки — на **одном** слое (например весь TM_OCH).
2. В рамку выбора — **только** линии и текст **этой** таблицы (без длинных «хвостов» линий).
3. В колонке **«Поз.»** — чистые номера `1`, `2`, `3`… (не только в «Наименовании»).
4. После **ЗАПУСТИТЬ** в палитре есть те же марки и количества.
5. Длина отрезков Line в таблице: на больших промышленных спецификациях линии длинные; на учебной маленькой таблице порог 5000 при выборе слоя на других чертежах может вести себя иначе — смотреть состав рамки.

### Связь с СПДС / proxy (отдельно от этой таблицы)

Уведомление **SPDSExtApp** при открытии DWG влияет на **выноски** (подсчёт в палитре), не на построение сетки из Line. Ручная таблица из Line — отдельный кейс: здесь ломается связка **линии → сетка → ColQty → пустая Поз. → фиксированная высота 250**.
